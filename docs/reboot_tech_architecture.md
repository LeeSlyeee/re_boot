> **작성일**: 2026-03-13

## 1. 전체 시스템 아키텍처

### 1.1 3-Tier 구조

Re:Boot은 클래식한 3-Tier 아키텍처 위에 AI 파이프라인을 결합한 구조이다.

**Presentation Layer (프론트엔드):**
- 학습자 앱: Vue.js 3 + Pinia + Vite 4 (포트 5173)
- 강사 대시보드: Vue.js 3 + Vite 5 (포트 5174)
- 두 앱 모두 SPA(Single Page Application) 구조이며, Vue Router + Navigation Guard로 인증 보호

**Application Layer (백엔드):**
- Django REST Framework (DRF) 기반
- SimpleJWT를 통한 JWT 인증
- 5개 Django 앱: users, learning, career, courses, analytics

**Data Layer (데이터):**
- MariaDB: 관계형 데이터 (사용자, 강의, 퀴즈, 포트폴리오)
- ChromaDB: 벡터 DB (RAG용 학습 콘텐츠 임베딩)

### 1.2 DB 모델 구조

**users 앱:**
- User: Role 기반 접근 제어 (STUDENT/INSTRUCTOR/MANAGER)
- ClassGroup: 반(클래스) 관리
- Enrollment: 학생-강의 수강 등록 (입장 코드 기반)

**learning 앱:**
- Lecture: 강의 정보
- Syllabus: 실라버스 (주차별 학습 계획)
- LearningObjective: 학습 목표 (체크리스트 항목)
- StudentChecklist: 학생별 체크리스트 완료 상태
- LearningSession: 학습 세션 (출석 + 진행 상태)
- STTLog: 실시간 STT 텍스트 청크 저장
- SessionSummary: AI 요약 텍스트 저장
- DailyQuiz: 일일 퀴즈 (AI 자동 출제)
- QuizQuestion: 퀴즈 문항
- QuizAttempt: 퀴즈 응시 기록
- AttemptDetail: 문항별 상세 응답
- VectorStore: ChromaDB 연동 벡터 인덱스 관리

**career 앱:**
- Portfolio: 듀얼 아웃풋 포트폴리오 (취업/창업 모드)
- MockInterview: 모의면접 세션
- InterviewExchange: 면접 질의응답 + 채점 결과

**courses 앱:**
- CourseSection: 코스 섹션 관리

## 2. STT 엔진 v3: 핵심 기술 파이프라인

### 2.1 음성 → 텍스트 변환 흐름

Re:Boot의 핵심 기술 차별점은 "실시간 강의 음성을 AI가 자동으로 필기하는" STT(Speech-to-Text) 파이프라인이다.

**전체 흐름:**

1. 학생이 학습 세션을 시작하면, 브라우저의 MediaRecorder API가 마이크 입력을 캡처한다.
2. 1분 단위로 오디오 청크를 분할한다 (Whisper Chunking).
3. 각 청크는 WebSocket 또는 HTTP로 백엔드에 전송된다.
4. 백엔드에서 OpenAI Whisper API를 호출하여 텍스트 변환을 수행한다.
5. 변환된 텍스트는 STTLog 테이블에 타임스탬프와 함께 저장된다.
6. Sentence Buffering: 불완전한 문장은 다음 청크와 병합하여 문장 단위 완성을 보장한다.

### 2.2 쉬는 시간 자동 요약

강의 중간 쉬는 시간(또는 세션 종료)에 자동 요약이 트리거된다.

1. 해당 세션의 모든 STTLog를 시간순으로 합친다.
2. GPT-4o에게 학습 맥락(실라버스 + 학습 목표)과 함께 요약 프롬프트를 전송한다.
3. AI가 요약 텍스트 + 핵심 키워드 + "확인해야 할 개념" 체크리스트를 생성한다.
4. 결과는 SessionSummary 테이블에 저장된다.
5. 동시에 요약 텍스트를 ChromaDB에 벡터 임베딩으로 저장 (RAG용).

### 2.3 IP 보호 전략 (지적재산권)

- 강사의 원본 음성 파일은 서버에 영구 저장하지 않는다.
- STT 처리 후 원본 오디오는 즉시 파기한다.
- AI가 재구성한 2차 저작물(요약 노트, 체크리스트)만 학생에게 제공한다.
- 이 정책은 기획 단계부터 의도적으로 설계된 것이며, 강사의 지적재산권을 존중하면서도 학습 효율을 극대화하는 상생 모델이다.

## 3. RAG(Retrieval-Augmented Generation) 기반 AI 튜터

### 3.1 동작 원리

Re:Boot의 AI 튜터는 범용 ChatGPT가 아니다. "오늘 수업에서 배운 내용"을 맥락(Context)으로 기억하는 Context-Aware 챗봇이다.

1. 학생이 질문을 입력한다. 예: "아까 강사님이 말한 비동기 처리 예시 다시 설명해줘"
2. 질문을 OpenAI Embedding API로 벡터화한다.
3. ChromaDB에서 해당 세션의 STT 요약 중 코사인 유사도가 높은 상위 5개 청크를 검색한다.
4. 검색된 학습 맥락 + 학생 질문 + 시스템 프롬프트를 GPT-4o에게 전송한다.
5. AI가 "수업 내용을 기반으로" 답변을 생성한다.

### 3.2 벡터 인덱싱 구조

- 각 STT 요약은 세션 ID, 강의 ID, 생성 시각과 함께 메타데이터로 태깅된다.
- 학생 질문 시 "현재 수강 중인 강의"의 벡터만 검색 범위에 포함시킨다 (Scoped Search).
- 이를 통해 다른 강의의 내용이 오답으로 혼입되는 것을 방지한다.

## 4. AI 퀴즈 자동 출제 시스템

### 4.1 출제 흐름

1. 학습 세션이 종료되면, 해당 세션의 요약 텍스트를 GPT-4o에 전송한다.
2. 프롬프트 지시: "이 수업 내용을 기반으로 객관식 5문항 + 주관식 2문항을 출제하라."
3. AI가 JSON 구조로 퀴즈를 반환한다 (질문, 선택지, 정답, 해설, 난이도, 해당 학습목표 매핑).
4. DailyQuiz + QuizQuestion 테이블에 저장한다.

### 4.2 채점 및 성취도 분석

- 객관식: 자동 채점
- 주관식: GPT-4o 기반 유사도 채점 (0-100점)
- 문항별로 연결된 LearningObjective를 추적하여 "어떤 학습 목표에서 부진한지" 데이터를 생성한다.
- 이 데이터가 리라우팅 엔진과 강사 대시보드에 공급된다.

## 5. 동적 리라우팅 엔진 (설계 완료, 알고리즘 구현 중)

### 5.1 개념

자동차 내비게이션의 "경로 재탐색" 기능을 학습에 적용한 것이다.

기존 LMS는 정해진 커리큘럼이 고정되어 있다. 학습자가 뒤처지면 "미수강"으로 표시될 뿐, 대안을 제시하지 않는다. Re:Boot의 리라우팅은 학습자의 현재 위치와 남은 기간을 분석하여 "지금부터 목표에 도달할 수 있는 최적 경로"를 자동 생성한다.

### 5.2 트리거 조건

- 7일 연속 미접속
- 퀴즈 3연속 50% 미만
- 세션 2주 이상 미완료
- 학습 목표 체크리스트 완료율 40% 미만

### 5.3 대응 로직

1. 남은 기간과 남은 학습 목표를 계산한다.
2. 학습 목표를 "필수"와 "선택"으로 분류한다.
3. 필수 목표만으로 "압축 패스트 트랙" 커리큘럼을 재구성한다.
4. 놓친 부분을 메우기 위한 맞춤형 콘텐츠(3분 요약 노트, 참조 링크)를 자동 배정한다.
5. 학습자에게 "새로운 경로를 찾았습니다" 알림을 전송한다.

## 6. AI 모의면접 시스템

### 6.1 6종 면접관 페르소나

| 코드 | 면접관 | 스타일 |
|:-----|:------|:-------|
| TECH_LEAD | 시니어 개발자 | 기술적 깊이 집요 추적, "Why" 집중, 대안 기술 비교 |
| FRIENDLY_SENIOR | 친절한 사수 | 편안한 분위기, 성장 가능성과 팀 적응력 확인 |
| HR_MANAGER | 인사 담당자 | 성격, 가치관, 갈등 해결 등 소프트 스킬 집중 |
| STARTUP_CEO | 스타트업 대표 | 속도, 비즈니스 가치, 주도적 문제 해결 |
| BIG_TECH | 빅테크 면접관 | CS 기초, 자료구조/알고리즘, 대규모 트래픽 |
| PRESSURE | 압박 면접관 | 허점 지적, 곤란한 상황 대처력 테스트 |

### 6.2 4차원 평가 Rubric

모의면접은 4개 차원으로 채점된다:

1. **기술적 깊이 (technical_depth)**: 개념 정확성, 기술 선택 이유, 대안 비교
2. **논리적 일관성 (logical_coherence)**: 답변 구조, STAR 기법 준수, 인과관계
3. **소통 능력 (communication)**: 설명 명확성, 구체적 예시 활용, 질문 의도 파악
4. **문제 해결력 (problem_solving)**: 실무 적용 능력, 예외 케이스 고려, 창의적 접근

전체 종합 점수는 4개 차원의 산술 평균에 ±5점 가중 조정을 적용한다.

### 6.3 AI 기본법 컴플라이언스

한국 AI 기본법(2026.01.22 시행)에 대해 Re:Boot은 다음을 이미 구현했다:
- AI 활용 사전 고지: "이 서비스는 AI를 활용합니다"
- AI 생성물 표시: 면접 질문/평가에 "AI 생성 콘텐츠" 라벨
- Disclaimer: "본 평가는 AI 자동 생성 참고 자료이며, 전문가 검증을 거치지 않았습니다"
- 비결정성 고지: "이 결과는 실제 채용 결과와 무관합니다"

Re:Boot의 AI 모의면접은 고영향 AI에 해당할 가능성이 낮다. 실제 채용 합격 여부를 결정하지 않고, 학습 보조 목적의 참고용 피드백만 제공하기 때문이다. 그러나 생성형 AI 관련 투명성 의무(사전 고지, 생성물 표시)는 적용 가능성이 있어 사전 대응을 완료했다.

## 7. 배포 아키텍처 (OCI)

### 7.1 배포 구조

- **서버**: Oracle Cloud Infrastructure (OCI) 컴퓨트 인스턴스
- **웹서버**: Nginx (리버스 프록시 + HTTPS)
- **백엔드**: Gunicorn + Django
- **프론트엔드**: Nginx에서 Vue.js 빌드 결과물 정적 서빙
- **도메인**: HTTPS를 통한 보안 통신

### 7.2 배포 최적화 패턴

1. **tar.gz 배포**: SSH 전송 속도 문제를 해결하기 위해, 로컬에서 빌드 결과물을 tar.gz로 압축 후 서버에서 해제하는 방식을 사용한다.
2. **Nginx path-to-alias 매핑**: Vue SPA의 라우팅을 Nginx의 try_files 지시어로 처리한다.
3. **Hardware-less Simulation**: 물리적 오디오 입력 없이 AI 파이프라인(STT→요약→퀴즈)을 테스트할 수 있는 시뮬레이션 모드를 구현했다.

### 7.3 보안

- JWT 토큰 기반 인증 (SimpleJWT)
- Vue Router Navigation Guard: 모든 인증 필요 페이지에 전역 가드 적용
- 404 Catch-All 라우팅: 존재하지 않는 경로 접근 시 브랜디드 404 페이지로 리다이렉트
- 역할 기반 접근 제어(RBAC): STUDENT/INSTRUCTOR/MANAGER별 API 권한 분리

## 8. 통합 테스트 결과 요약 (Phase 1-8)

Re:Boot은 8단계 통합 테스트를 순차적으로 수행하며 기능 검증을 완료했다:

- Phase 1: 기본 인증 및 RBAC 테스트
- Phase 2: STT 파이프라인 + 요약 생성 테스트
- Phase 3: 퀴즈 자동 출제 및 채점 테스트
- Phase 4: 체크리스트 CRUD + 학습 진행률 테스트
- Phase 5: 포트폴리오 생성 (취업/창업 듀얼 모드)
- Phase 6: 모의면접 (6종 페르소나 + 4차원 채점)
- Phase 7: 강사 대시보드 통합 테스트
- Phase 8: OCI 프로덕션 배포 + E2E 테스트

주요 결과: STT 요약 품질 검증, 퀴즈 정답률 데이터 수집, 포트폴리오 PDF 생성 확인, 모의면접 결과 리포트 화면 검증, OCI 환경에서의 전체 흐름 확인 완료.
