# 🚀 Re:Boot 프로젝트 역할 및 실행 전략

## 1. 프로젝트 내 담당 역할 (추천)
**On-Device AI / Edge Inference 엔지니어 (설계 및 구현)**

*   **멘토 추천 방향 매칭:**
    *   `ML model → hardware mapping` (하드웨어 특성에 맞춘 ML 오프로딩 아키텍처)
    *   `Quantization / pruning → HW` (로컬 기기 구동을 위한 모델 경량화)
*   **핵심 연관 기능:** Phase 0-3 직속 퀴즈 발동을 위한 **엣지 기반 키워드 스팟팅(KWS) 및 로컬 STT 파이프라인 구축**

---

## 2. 💡 아키텍처 설계: 하드웨어 한계 극복 전략
보유하고 있는 라즈베리파이 3 B+의 성능적 한계(1GB RAM, 연산력 부족)를 인정하고, 이를 해결하기 위해 **역할을 분담하는 분산 아키텍처(Layered Architecture)**를 채택합니다.

### ① Edge (Raspberry Pi 3 B+) : 스마트 트리거 (KWS)
*   **역할:** 항시 켜져 있는 상태에서 "잠깐 퀴즈!" 또는 "이해하셨나요?" 같은 트리거 키워드만 감지만 수행.
*   **기술/역량:** 수 MB 크기의 초경량 오디오 분류(KWS) 모델 학습. (시간적 여유 허용 시 INT8 양자화 적용 및 C/C++ 기반 맵핑 시도).
*   **동작:** 감지 성공 시, 즉각적으로 교수자 PC에 소켓 통신을 통해 `TRIGGER_ON` 신호 전송.

### ② Local Host (교수자 노트북) : 무거운 STT 및 LLM 오프로딩 (Offloading)
*   **역할:** 라즈베리파이의 신호를 받으면 구동되어 실제 퀴즈 음성을 텍스트로 바꾸고(STT), AI를 통해 퀴즈를 생성(LLM).
*   **기술/역량:** `Whisper.cpp` 또는 경량화된 로컬 STT 엔진을 사용해 텍스트 추출 후, 프롬프트 엔지니어링을 통해 ChatGPT API (JSON 출력 강제) 연동.
*   **동작:** 추출된 퀴즈와 정답 데이터를 백엔드/웹으로 전달.

## 3. 🎯 포트폴리오 어필 포인트 (면접관의 시선)
단순한 웹 백엔드 개발에 비해 **신입/주니어 레벨에서 극도로 희소한 기술적 깊이**를 보여줄 수 있습니다.

1.  **현실 인지:** 무조건 최신 API만 가져다 쓰는 것이 아니라 하드웨어(라즈베리파이 3 B+)의 병목과 제약을 인지함.
2.  **구조 설계 능력:** 한계를 극복하기 위해 KWS와 단기 STT를 분리하는 로지컬한 시스템 아키텍트를 본인 스스로 설계함.
3.  **Low-level ↔ High-level 통합:** 하드웨어 단의 모델 맵핑(C/C++, TFLite)부터 LLM을 활용한 프롬프트 엔지니어링 파이프라인(Python API 연동)까지 풀스택 인프라 구축.

## 4. 🔗 시스템 아키텍처 상세 (데이터 흐름 및 통신)
이 시스템은 단순한 기능 구현을 넘어, 효율성과 프라이버시, 예산 절감을 동시에 달성하는 매우 견고한 인프라를 자랑합니다.

### ① 실시간 데이터 Flow (마이크 입력부터 학생 화면 퀴즈까지)
1.  **[수음 & KWS] Edge:** 라즈베리파이 마이크가 대기하다가 "이해하셨나요?" 등의 트리거 키워드 감지.
2.  **[신호 송신] Network:** 라즈베리파이가 로컬 Wi-Fi망을 통해 교수자 PC로 소켓 신호(`TRIGGER_ON`) 발송. (오디오 전송 아님)
3.  **[수음 & STT] PC:** PC 측 마이크 활성화 및 실제 강의 내용 녹음 → 로컬 엔진(`Whisper.cpp`)으로 텍스트 추출.
4.  **[퀴즈 생성] LLM:** 추출된 '텍스트'만 ChatGPT API로 전송하여 퀴즈 데이터(JSON) 획득.
5.  **[배포] Web:** 수신된 JSON을 DB에 저장 후, 실시간 통신(웹소켓 등)을 거쳐 학생들 기기에 즉각 배포.

### ② 하드웨어 제어 기반 통신 환경 (Wi-Fi 인트라넷)
*   라즈베리파이와 교수자 PC는 **동일 강의실 내 Wi-Fi(로컬망)**를 통한 TCP/UDP 소켓 통신을 이용합니다.
*   외부 인터넷망 트래픽을 타지 않으므로 밀리초(ms) 단위의 **Zero-Latency** 통신을 보장하며, 현장 인터넷망이 잠시 불안정해도 라즈베리파이-PC 간 KWS 호출(신호 교환)은 쾌적하게 동작합니다.

### ③ 통상적 API 사용 대비 하드웨어 매핑 아키텍처의 비교 우위 (Merit)
*   **유지비용(API 과금) 최적화:** 외부 실시간 STT API(Google STT 등)를 2시간 내내 열어둘 경우 막대한 오디오 스트리밍 과금이 발생합니다. 본 구조는 음성 처리를 현장 하드웨어(PC)에서 무료로 끝내고, 텍스트 단 몇 줄만 LLM으로 보내므로 **서버 유지비를 1/100 수준으로 획기적으로 절감**합니다.
*   **강력한 프라이버시 보호:** 강의 원본 음성이 외부 서버로 한 번도 송출되지 않고 로컬에서 파기(처리)되므로, 철저한 지적재산권 보호 요건을 100% 충족합니다.
*   **망 부하(Bandwidth) 최소화:** 무거운 실시간 오디오 스트림을 쏘지 않으므로, 수십 명의 수강생이 동시 접속하여 인터넷망 부하가 심한 부트캠프 환경에서도 시스템 뻗음/지연 없이 안정적인 서비스(Robustness)가 가능합니다.

## 5. ⏳ 3주 단기 구현 마일스톤 및 생존 전략
모든 것을 바닥(C/C++)부터 만들면 일정 내 완수가 불가능합니다. 파이프라인을 관통시키는 것을 최우선 목표로 잡습니다.

*   **[1주 차] 파이프라인 관통 및 KWS 생존:**
    *   라즈베리파이 마이크 수음 환경 세팅.
    *   Float32 기반의 아주 가벼운 KWS 모델을 **Python 환경**(`tflite_runtime`)으로 우선 구동.
    *   트리거 감지 시 교수자 PC로 TCP/UDP 소켓 통신 신호 송신 확인.
*   **[2주 차] STT 및 LLM 연동:**
    *   교수자 PC에서 마이크 분리 수음 환경 세팅.
    *   신호 수신 시 로컬 STT 실행 및 텍스트 추출 확인.
    *   ChatGPT API를 통해 **정해진 JSON 포맷**으로 퀴즈 데이터 응답받기 성공 확인.
*   **[3주 차] 모델 최적화 및 안정화 (보너스 목표):**
    *   KWS 모델 양자화(INT8) 적용 및 정확도 테스트.
    *   (여유 시) 라즈베리파이 실행 환경을 Python에서 C/C++로 포팅 시도.
    *   예외 처리 (마이크 에러, LLM 응답 포맷 에러 등) 방어 코드 작성 및 프론트 연동 테스트.
