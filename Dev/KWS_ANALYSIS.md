# KWS(Keyword Spotting) 시스템 오디오 파이프라인 분석 및 개선 리포트

본 문서는 실시간 KWS 감지 성능 저하 문제를 해결하기 위해 수행된 분석 내용과 단계별 개선 사항, 그리고 현재 진행 중인 모델 인식률 최적화 과정을 정리합니다.

---

## 1. 주요 결함 및 해결 완료 사항

### 1-1. 프론트엔드 오디오 데이터 유실 (SOLVED ✅)
- **증상**: 매초 약 2.4%(384 샘플)의 오디오 데이터가 유실되어 음성 파형이 불연속적으로 깨짐.
- **원인**: `ScriptProcessorNode`(4,096)와 전송 버퍼(16,000) 크기 불일치 시 잔여 샘플을 버리는 로직 오류.
- **해결**: **이월(Carry-over) 버퍼 방식** 도입. 유입되는 모든 샘플을 순차적으로 채우고, 전송 후 남은 샘플부터 다음 버퍼를 다시 구성하여 0% 유실 달성.

### 1-2. Base64 인코딩 오버헤드 (SOLVED ✅)
- **증상**: 문자열 루프 기반 인코딩으로 인해 높은 CPU 점유율 및 전송 지연 발생.
- **해결**: `String.fromCharCode.apply` 및 `ArrayBuffer` 직접 접근 방식을 통한 **고속 인코딩 최적화** 적용.

### 1-3. 실시간 응답성 결여 (SOLVED ✅)
- **증상**: 1초 분량의 오디오가 모두 모일 때까지 기다린 후 전송하여 감지 지연 발생.
- **해결**: 전송 단위를 **0.25초(4,000 샘플)**로 단축. 백엔드 `kws_engine.py`가 가변 길이 청크를 지원하도록 수정하여 즉각적인 피드백 구조 구축.

### 1-4. 오디오 신호 전처리 노이즈 (SOLVED ✅)
- **증상**: 브라우저의 에코 제거, 소음 억제 기능이 KWS 모델에 필요한 음성 특징(MFCC)을 변형시킴.
- **해결**: `getUserMedia` 설정에서 `echoCancellation`, `noiseSuppression`, `autoGainControl`을 모두 **false**로 설정하여 순수 원본 음성 확보.

## 2. 현재 분석 중인 잔류 이슈: 인식률 저하 (Unknown 100%)

### 2-1. MFCC 추출 사양 불일치 의혹 (HIGH PRIORITY 🚨)
- **가설**: 학습 시 사용된 TF 컨트리뷰션 오디오 엔진의 MFCC 파라미터(예: Window 30ms / Stride 10ms)와 현재 추론 서버 설정(40ms / 20ms)이 다를 가능성.
- **현상**: 신호 품질 개선 후에도 모델이 모든 입력을 `_unknown_`으로 판단함.

### 2-2. 입력 신호 크기 및 편향 문제 (PROPOSED FIX 🛠️)
- **개선 내용**: 
    - **피크 정규화(Peak Normalization)**: 입력 볼륨이 작아도 항상 0.9 Peak로 증폭하여 모델 입력 범위 최적화.
    - **DC Offset 제거**: 마이크 특성으로 인한 신호의 수직 편향을 제거하여 정확한 주파수 분석 유도.

## 3. 정밀 진단 모드 (Diagnostic Mode)

모델의 '속마음'을 보기 위해 라즈베리파이 서버에 다음 기능을 추가했습니다:
- **Top-3 확률 출력**: 단순히 1위만 보는 게 아니라, `quiz`나 `understand` 라벨의 점수 변화를 실시간으로 모니터링.
- **스무딩 최소화**: 윈도우 스무딩을 제거(`maxlen=1`)하여 짧은 감지 순간도 놓치지 않고 로그로 기록.
