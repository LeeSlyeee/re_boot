# 🚀 Phase 3: 키워드 감지 기반 팝업 프로세스 설계안

## 1. 개요
현재 라즈베리 파이(Edge)에서 KWS 모델이 "퀴즈(QUIZ)"나 "이해했나요(UNDERSTAND)" 키워드를 감지하고, HTTP 웹훅을 통해 백엔드(Django)로 신호를 보내는 흐름이 완성되었습니다.
본 설계안은 백엔드에서 받은 키워드 신호를 바탕으로 교수자 및 학생 화면에 알맞은 팝업 이벤트(UI)를 띄우는 프론트엔드/백엔드 통신 플로우를 정의합니다.

## 2. 키워드별 동작 정의

### 🎯 키워드 1: "퀴즈 (QUIZ)"
*   **교수자 의도**: 강의 도중 학생들의 이해도를 점검하기 위해 퀴즈를 내려고 함.
*   **현재 백엔드 동작**: `/kws-webhook/`에서 'QUIZ' 수신 시, 백엔드 스레드에서 AI를 통해 STT 기반으로 퀴즈를 자동 생성.
*   **신규 UI 프로세스 (교수자 화면)**:
    1. 팝업 노출: "🤖 '퀴즈' 키워드가 감지되었습니다! AI가 퀴즈를 생성하고 있습니다..." (로딩 표시)
    2. 팝업 업데이트(생성 완료 직후): 생성된 퀴즈 내용(문제, 보기, 정답)과 함께 [✅ 학생들에게 발송하기] / [❌ 취소] 버튼 제공.
*   **신규 UI 프로세스 (학생 화면)**:
    1. 교수자가 퀴즈를 승인(발동)하면 폴링을 통해 새로운 활성 퀴즈를 감지하고 퀴즈 풀이 팝업 노출.

### ❓ 키워드 2: "이해했나요 (UNDERSTAND)"
*   **교수자 의도**: 특정 개념 설명 후 학생들이 제대로 이해했는지 확인(Pulse Check)하려고 함.
*   **신규 UI 프로세스 (교수자 화면)**:
    1. 팝업 노출: "👀 '이해했나요' 키워드가 감지되었습니다. 학생들에게 이해도 조사를 요청합니다." (자동 전송 혹은 수동 확인 버튼 제공)
*   **신규 UI 프로세스 (학생 화면)**:
    1. 서버로부터 '이해도 조사(Pulse Check)' 이벤트 수신.
    2. 팝업 노출: "이번 설명, 잘 이해되셨나요?"
       - [✅ 이해했어요] / [❓ 헷갈려요 (지원 요청)] 버튼 제공
    3. 응답 결과는 기존 이해도 펄스(`PulseCheck`) 데이터베이스에 기록되어 교수의 실시간 그래프에 반영.

## 3. 통신 아키텍처 (Polling 기반)
현재 WebSocket 대신 Polling(폴링)을 사용하고 있으므로, 상태 기반 플로우를 유지합니다.

1. **상태 관리 (Backend)**
   - `LiveSession` 모델 혹은 Redis 캐시에 최근 감지된 키워드 이벤트(Event) 저장 (예: `last_keyword_event: {type: 'QUIZ', timestamp: ...}`).
2. **학생/교수 브라우저 폴링 (Frontend)**
   - 기존의 상태 조회 API(`GET /api/learning/live/{id}/status/` 등)에 `recent_kws_event` 필드를 추가하여 일정 간격으로 확인.
   - 브라우저가 새로운 이벤트를 감지하면 해당 이벤트 처리를 수행하고 로컬 상태 업데이트.
